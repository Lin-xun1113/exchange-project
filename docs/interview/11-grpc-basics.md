# gRPC 基础入门：什么是服务端和客户端

## 前置概念：分布式系统里的"打电话"

### 现实世界的类比

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           餐厅点餐系统                                      │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   顾客（发起请求）                                                         │
│       │                                                                    │
│       │  "我要一份宫保鸡丁"                                                │
│       ▼                                                                    │
│   服务员（接收请求，处理，返回结果）                                         │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘

在 gRPC 里：
  - 顾客 = 客户端（Client）
  - 服务员 = 服务端（Server）
  - "我要一份宫保鸡丁" = 请求（Request）
  - "好的，稍等" = 响应（Response）
```

## 什么是 gRPC？

### 定义

gRPC 是 Google 开发的高性能远程过程调用框架。

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                              gRPC 特点                                      │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   1. 基于 HTTP/2：支持多路复用、双向流、头部压缩                             │
│   2. Protocol Buffers：高效的序列化协议，比 JSON 快 5-10 倍                 │
│   3. 多语言支持：Go、Java、Python、C++ 等                                  │
│   4. 双向流：客户端和服务端可以同时发送数据                                  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 对比 REST API

| 维度 | REST API | gRPC |
|------|----------|------|
| 协议 | HTTP/1.1 | HTTP/2 |
| 数据格式 | JSON | Protocol Buffers |
| 传输效率 | 较低 | 高 5-10 倍 |
| 代码生成 | 无 | 自动生成强类型客户端/服务端 |
| 流式支持 | 需要额外处理 | 原生支持 |
| 浏览器支持 | 天然支持 | 需要 grpc-web |

## 服务端（Server）vs 客户端（Client）

### 最简单的理解

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                              核心区别                                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   服务端（Server）：                                                        │
│   - 守在某个地址（IP:端口）等着                                             │
│   - 有人来敲门，就处理请求                                                   │
│   - 处理完返回结果                                                         │
│                                                                             │
│   客户端（Client）：                                                        │
│   - 主动去找服务端连接                                                     │
│   - 发送请求                                                               │
│   - 等待响应                                                               │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 类比：打电话

```
   客服中心（Server）
   │
   │  24 小时在线，等你打电话
   │
用户 ──拨打电话──► 客服中心
   │                    │
   │                    │ 接听，处理问题
   │                    │
   ◄──回答问题─── 用户

- Server 守在电话旁（监听端口）
- Client 主动拨号（发起连接）
```

### 代码层面的区别

#### 服务端代码

```go
// internal/order/server.go（订单服务）
type OrderServer struct {
    // 存放订单数据
}

func (s *OrderServer) CreateOrder(ctx context.Context, req *pb.CreateOrderRequest) (*pb.CreateOrderResponse, error) {
    // 处理创建订单的逻辑
    order := &pb.Order{
        Id:      generateID(),
        UserId:  req.UserId,
        Amount:  req.Amount,
    }
    return &pb.CreateOrderResponse{Order: order}, nil
}

// 启动服务（监听 50051 端口）
lis, _ := net.Listen("tcp", ":50051")
grpcServer := grpc.NewServer()
pb.RegisterOrderServiceServer(grpcServer, &OrderServer{})
grpcServer.Serve(lis)
```

#### 客户端代码

```go
// internal/gateway/client/order_client.go（网关调用订单服务）
type OrderClient struct {
    conn   *grpc.ClientConn
    client pb.OrderServiceClient  // 自动生成的客户端
}

func NewOrderClient(addr string) (*OrderClient, error) {
    // 1. 连接到服务端（就像拨号）
    conn, err := grpc.Dial(addr, grpc.WithInsecure())
    if err != nil {
        return nil, err
    }

    // 2. 创建客户端（就像拿到电话听筒）
    client := pb.NewOrderServiceClient(conn)

    return &OrderClient{conn: conn, client: client}, nil
}

func (c *OrderClient) CreateOrder(ctx context.Context, userId string, amount float64) (*pb.Order, error) {
    // 3. 发送请求（说话）
    resp, err := c.client.CreateOrder(ctx, &pb.CreateOrderRequest{
        UserId: userId,
        Amount: amount,
    })
    if err != nil {
        return nil, err
    }
    return resp.Order, nil  // 4. 收到响应（听回答）
}
```

## 为什么有两个 OrderClient？

### 问题回放

```
internal/gateway/client/order_client.go      ← Gateway 用
internal/matching/client/order_client.go     ← Matching 用
```

### 解释

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                          两个 OrderClient 的原因                            │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   订单服务（Order Service）是同一个                                          │
│   但调用它的"角色"不同                                                       │
│                                                                             │
│   ═══════════════════════════════════════════════════════════════════════   │
│                                                                             │
│   Gateway 的 OrderClient：                                                   │
│   ┌─────────────────────────────────────────────────────────────────┐       │
│   │ 场景：用户下单，Gateway 调用 Order Service 记录订单              │       │
│   │                                                                   │       │
│   │ 策略：熔断器保护                                                  │       │
│   │ - 如果 Order Service 挂了                                        │       │
│   │ - Gateway 快速失败，返回 503 给用户                              │       │
│   │ - 而不是一直等待导致自己也被拖垮                                  │       │
│   └─────────────────────────────────────────────────────────────────┘       │
│                                                                             │
│   ═══════════════════════════════════════════════════════════════════════   │
│                                                                             │
│   Matching 的 OrderClient：                                                  │
│   ┌─────────────────────────────────────────────────────────────────┐       │
│   │ 场景：撮合成功，Matching 调用 Order Service 写订单               │       │
│   │                                                                   │       │
│   │ 策略：重试保护                                                    │       │
│   │ - 如果 Order Service 临时抖动                                    │       │
│   │ - Matching 等一等、重试几次                                      │       │
│   │ - 订单必须写进去，不能丢                                          │       │
│   └─────────────────────────────────────────────────────────────────┘       │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 拦截器对比

| 拦截器 | Gateway | Matching | 原因 |
|--------|---------|----------|------|
| RequestID | ✅ | ✅ | 都需要链路追踪 |
| **熔断器** | ✅ | ❌ | Gateway 要快速失败 |
| **重试** | ✅ | ✅ | 但时机不同 |
| 日志 | ✅ | ✅ | 都需要日志 |
| 超时 | ✅ | ✅ | 都防止卡死 |

## 拦截器链详解

### 什么是拦截器？

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                              拦截器原理                                      │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   拦截器就像"流水线上的质检员"                                               │
│                                                                             │
│   请求                                                                     
│     │                                                                      
│     ▼                                                                      
│   ┌────────────────────────────────────────────────────────────────────┐   │
│   │ 质检员 A（加 request_id）                                           │   │
│   └────────────────────────────────────────────────────────────────────┘   │
│     │                                                                      
│     ▼                                                                      
│   ┌────────────────────────────────────────────────────────────────────┐   │
│   │ 质检员 B（熔断保护）                                               │   │
│   └────────────────────────────────────────────────────────────────────┘   │
│     │                                                                      
│     ▼                                                                      
│   ┌────────────────────────────────────────────────────────────────────┐   │
│   │ 质检员 C（记录日志）                                               │   │
│   └────────────────────────────────────────────────────────────────────┘   │
│     │                                                                      
│     ▼                                                                      
│   实际处理                                                                │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 项目代码中的拦截器链

```go
// internal/gateway/client/order_client.go

conn, err := grpc.Dial(addr,
    grpc.WithTransportCredentials(insecure.NewCredentials()),
    grpc.WithStatsHandler(otelgrpc.NewClientHandler()),  // 链路追踪
    grpc.WithChainUnaryInterceptor(
        grpcx.UnaryClientRequestID(),           // ① 加 request_id
        circuitBreakerInterceptor(cb, "order"),  // ② 熔断器
        logInterceptor,                         // ③ 日志
        timeout.UnaryClientInterceptor(),       // ④ 超时
        retry.UnaryClientInterceptor(),         // ⑤ 重试
    ),
)
```

### 每个拦截器的作用

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                              拦截器职责表                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   拦截器                    │ 作用                    │ 简单理解             │
│   ─────────────────────────────────────────────────────────────────────── │
│   UnaryClientRequestID()    │ 塞 request_id           │ "我是谁"             │
│   circuitBreakerInterceptor │ 熔断保护                │ "别烫手"             │
│   logInterceptor           │ 记录日志                │ "记下来"             │
│   timeout                   │ 超时控制                │ "别等太久"           │
│   retry                     │ 重试机制                │ "再试一次"           │
│   otelgrpc.NewClientHandler │ 链路追踪                │ "画链路图"           │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 面试高频问题

### Q1: 解释一下 gRPC 的服务端和客户端？

**回答要点**：
```
服务端（Server）：
- 被动等待，监听某个端口
- 接收请求、处理业务逻辑、返回响应
- 类似于餐厅服务员

客户端（Client）：
- 主动连接，指定服务端的地址
- 发送请求、等待响应
- 类似于餐厅顾客

在代码中：
- Server 定义接口实现，注册到 grpc.Server
- Client 通过 grpc.Dial 连接，拿到 Stub 调用方法
```

### Q2: gRPC 和 REST API 的区别？

**回答要点**：
| 对比项 | REST | gRPC |
|--------|------|------|
| 协议 | HTTP/1.1 | HTTP/2 |
| 格式 | JSON | Protobuf |
| 性能 | 一般 | 高 5-10 倍 |
| 代码生成 | 无 | 自动生成 |
| 流式 | 复杂 | 原生支持 |

### Q3: 拦截器链的执行顺序？

**回答要点**：
```
请求方向（先执行）：
UnaryClientRequestID → CircuitBreaker → Log → Timeout → Retry → 实际调用

响应方向（后执行）：
实际调用 → Retry → Timeout → Log → CircuitBreaker → UnaryClientRequestID
```

### Q4: 什么时候用熔断器，什么时候用重试？

**回答要点**：
```
熔断器（Gateway 用）：
- 保护自己不被打垮
- 适合：对外暴露的入口服务
- 效果：快速失败，返回错误

重试（Matching 用）：
- 确保请求必须成功
- 适合：写入类操作（订单、支付）
- 效果：等一等，再试几次

总结：
- Gateway = "挂了就算了" → 用熔断器
- Matching = "必须成功" → 用重试
```
